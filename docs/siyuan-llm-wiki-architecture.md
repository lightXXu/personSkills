# LLM Wiki 技术方案

## 仓库边界

本仓库是公开 GitHub 仓库，只维护可公开的 skill、脚本、模板和技术方案。LLM Wiki 的真实笔记内容不在本地 Markdown 目录长期维护，也不要提交到 GitHub；正式知识内容以思源笔记中的 `llm wiki` 笔记本为准。

## 方案定位

LLM Wiki 的终局不是传统 RAG，也不是一个更复杂的文件夹系统，而是一个运行在思源笔记之上的个人长期上下文系统。

它的目标是让新知识进入后，能够被持续提炼、连接、检索、审计，并在未来被人和 LLM 低成本复用。

一句话定义：

```text
以思源块系统为底座，用主题地图组织上下文，用概念卡沉淀判断，用来源卡保留证据，用项目卡驱动行动，并通过 attributes、refs、SQL 和可选语义检索形成可演化的 LLM 记忆层。
```

## 终局形态

### 终局系统长什么样

终局的 LLM Wiki 应该包含六层：

```text
输入层
  -> 加工层
  -> 知识层
  -> 索引层
  -> 检索层
  -> 协作层
```

各层职责如下：

| 层级 | 目标 | 典型内容 |
|---|---|---|
| 输入层 | 接住新信息 | 手动输入、网页剪藏、对话记录、读书笔记、项目记录 |
| 加工层 | 把原始信息变成可复用知识 | 摘要、概念抽取、判断提炼、来源保留、置信度标注 |
| 知识层 | 保存长期可读的知识对象 | 概念卡、来源卡、项目卡、问题卡、主题地图 |
| 索引层 | 让知识能被机器检索 | `blocks`、`attributes`、`refs`、FTS、可选向量索引 |
| 检索层 | 找到当前问题需要的上下文 | SQL 检索、反链检索、主题地图导航、可选语义检索 |
| 协作层 | 让 LLM 参与维护知识库 | 读取上下文、推荐插入位置、生成卡片草稿、人审核后写回 |

### 终局的数据对象

终局里不应该只有“文档”，而应该有几类稳定对象：

| 对象 | 保存什么 | 不保存什么 |
|---|---|---|
| 概念卡 | 可复用理解、判断、边界、适用场景 | 大段原文 |
| 来源卡 | 资料摘要、证据、出处、我是否认同 | 抽象方法论 |
| 项目卡 | 目标、当前阶段、行动计划、项目判断 | 泛泛知识 |
| 问题卡 | 未解决问题、待验证判断、研究路径 | 已经稳定的结论 |
| 主题地图 | 某个主题下的导航、必读卡片、当前共识 | 零散摘录 |
| 检索工作台 | 常用 SQL、卡片索引、审计入口 | 知识正文 |

### 终局的核心能力

终局系统至少要具备这些能力：

| 能力 | 说明 |
|---|---|
| 卡片索引 | 按类型、状态、关键词、置信度查询所有卡片 |
| 反链网络 | 查询某个概念被哪些来源、项目、地图引用 |
| 主题地图导航 | LLM 能根据地图快速构造上下文 |
| 新知识接入助手 | 自动检索旧上下文，建议更新或新建 |
| 周期性审计 | 找出低置信度、长期未更新、孤立卡片 |
| 冲突检测 | 发现同一概念下互相矛盾的判断 |
| 语义补充检索 | 卡片规模变大后，用向量搜索补充召回 |

### 终局的技术原则

1. 目录是入口，不是唯一索引。
2. 主题地图是 LLM 的导航页，不只是人工目录。
3. 概念卡保存判断，来源卡保存证据，项目卡保存行动上下文。
4. `attributes` 保存机器可检索分类。
5. `refs` 保存真实关系。
6. SQL 是第一优先级检索层。
7. 语义检索只作为增强，不替代结构化知识。
8. 自动化必须保留人工审核入口。

## 思源底座设计

思源笔记的核心不是文件夹，也不是纯 Markdown，而是块系统。它支持块级引用、双向链接、自定义属性、SQL 查询嵌入和数据库表格视图。

底层上，思源工作区把文档保存为 `.sy` JSON 文件；索引和查询能力通过内置数据库和 API 暴露。当前已验证可用于本系统的表包括：

| 表 | 用途 |
|---|---|
| `blocks` | 块内容、标题、路径、层级、更新时间 |
| `attributes` | 自定义属性，如 `custom-type`、`custom-status` |
| `refs` | 块引用和反链关系 |
| `spans` | 行内元素 |
| `blocks_fts*` | 全文索引相关表 |

官方参考：

- SiYuan README: https://github.com/siyuan-note/siyuan
- SiYuan API: https://github.com/siyuan-note/siyuan/blob/master/API.md

因此，LLM Wiki 不应该绕开思源另建一套知识库，而应该顺着思源原生能力演进：

```text
目录入口
  -> 块内容
  -> 自定义属性
  -> 块引用反链
  -> SQL 检索
  -> 主题地图导航
  -> LLM 辅助写回
```

## 从终局倒推当前设计

为了达到终局，不需要一开始就建设全部能力。当前最重要的是先搭出一个低成本、可持续、能被检索的最小系统。

当前设计如下：

```text
00 收集箱
10 概念卡
20 来源笔记
30 项目知识
40 长期问题
80 模板
90 主题地图
```

这些目录对应终局对象：

| 当前目录 | 对应终局对象 | 作用 |
|---|---|---|
| `00 收集箱` | 输入层 | 承接新材料和临时输入 |
| `10 概念卡` | 概念卡 | 沉淀可复用概念、判断、边界 |
| `20 来源笔记` | 来源卡 | 保存资料摘要、证据、出处 |
| `30 项目知识` | 项目卡 | 保存目标、行动、项目上下文 |
| `40 长期问题` | 问题卡 | 保存未决问题和待验证判断 |
| `80 模板` | 加工规则 | 固化卡片结构和字段 |
| `90 主题地图` | 导航层 | 做知识导航和 LLM 上下文入口 |

当前已经启用的底层能力：

| 能力 | 当前状态 | 用途 |
|---|---|---|
| `blocks` | 已可用 | 搜标题、正文、路径、层级 |
| `attributes` | 已启用 | 用 `custom-type` 等属性筛选卡片 |
| `refs` | 已启用 | 用 `((块ID "标题"))` 建立真实反链 |
| SQL 查询 | 已验证 | 可通过 `/api/query/sql` 查询 |
| 主题地图 | 已建立 | 作为上下文入口 |
| 模板 | 已升级 | 引导新卡带检索属性和引用关系 |

当前新增知识流程：

```text
新材料
  -> 00 收集箱
  -> 抽取 claims / keywords / sources / projects / open questions
  -> 检索 90 主题地图 / 10 概念卡 / 30 项目知识 / 40 长期问题 / 20 来源笔记
  -> 判断：更新旧卡 / 新建卡 / 标记 stale / 记录冲突说明
  -> 生成更新建议，等待用户确认
  -> 写入 attributes 和关键 refs
  -> 更新 90 主题地图或审计清单
```

## 当前设计评估

### 为什么当前设计是合适起点

当前设计是目前阶段的最优起步方案，原因是：

- 启动成本低，不需要插件、向量库或复杂自动化。
- 对人可读，对 LLM 也可读。
- 已经贴合思源原生能力，而不是另起一套外部数据库。
- 目录结构和终局对象是一一对应的，未来可以平滑演进。
- 已经开始用 `custom-*` 属性和真实块引用，后续 SQL 检索和反链分析有基础。

### 当前设计的边界

当前设计不是终局，主要限制是：

- 目录仍然承担较多组织职责，后期应逐步让属性、反链和主题地图承担更多检索职责。
- 新卡属性目前需要人工或 CLI 写入，尚未自动化。
- 主题地图还比较薄，未来需要从“链接列表”升级成“上下文导航页”。
- 还没有检索工作台，常用 SQL 没有沉淀成固定入口。
- 尚未使用语义检索；目前主要依靠 SQL、全文搜索和块引用。

## 演化路线

### 阶段 0：结构落地

状态：已完成。

目标：

- 创建 `llm wiki` 笔记本；
- 建立中文目录结构；
- 创建基础卡片和模板；
- 明确每日流程；
- 初步写入核心示例。

里程碑：

- `00 收集箱` 到 `90 主题地图` 的目录已存在；
- 至少有 1 张概念卡、1 张来源卡、1 张项目卡、1 张主题地图；
- README 说明清楚系统用途。

完成标准：

- 新材料知道放哪里；
- 旧材料知道如何抽卡；
- 不再纠结目录结构。

### 阶段 1：原生索引启用

状态：已基本完成。

目标：

- 在核心文档上写入 `custom-*` 属性；
- 使用块引用替代重要的 `[[标题]]` 链接；
- 验证 `attributes` 和 `refs` 能被 SQL 查询。

里程碑：

- `custom-type` 可区分 `concept`、`source`、`project`、`question`、`map`、`template`；
- `refs` 表里出现真实块引用记录；
- 核心主题地图引用概念、项目、来源、问题和方案。

完成标准：

- 能用 SQL 查所有概念卡；
- 能查某个概念的反链；
- 能根据 `custom-status` 找到草稿或稳定卡片。

### 阶段 2：检索工作台

目标：

把分散的 SQL 查询和检索习惯变成固定入口。

建议新增文档：

```text
90 主题地图 / 检索工作台
```

内容包括：

- 所有概念卡；
- 所有来源笔记；
- 所有进行中项目；
- 所有开放问题；
- 低置信度卡片；
- 最近更新卡片；
- 孤立卡片；
- 某个块的反链查询模板。

示例 SQL：

```sql
SELECT a.block_id, b.content, b.hpath
FROM attributes a
JOIN blocks b ON a.block_id = b.id
WHERE a.box = 'BOX_ID'
  AND a.name = 'custom-type'
  AND a.value = 'concept';
```

里程碑：

- `检索工作台` 页面创建完成；
- 至少沉淀 5 条常用 SQL；
- 每次新知识接入前，先走工作台检索。

完成标准：

- 不用临时想怎么搜；
- LLM 可以根据固定 SQL 查上下文；
- 人也能从一个入口看到系统状态。

### 阶段 3：稳定卡片生产

目标：

让系统从“结构搭好了”进入“持续产生高质量知识”的阶段。

工作方式：

- 每天处理 1-3 条新材料；
- 每周至少沉淀 3 张概念卡；
- 每周更新一次主题地图；
- 每周检查一次低置信度或孤立卡片。

里程碑：

- 概念卡达到 30 张；
- 来源笔记达到 30 张；
- 至少 5 张主题地图；
- 至少 5 个项目卡产生实际行动影响。

完成标准：

- 新知识不是堆在收集箱里；
- 旧卡会被更新，而不是只新建；
- 主题地图能承担 LLM 上下文入口。

### 阶段 4：半自动接入助手

目标：

让 LLM 帮你完成“检索旧上下文 -> 推荐插入位置 -> 生成草稿”的前半段，但写入仍由人确认。

流程：

```text
输入新材料
  -> LLM 提取关键词和判断
  -> SQL 搜索 blocks / attributes / refs
  -> 返回候选旧卡
  -> 判断更新旧卡还是新建
  -> 生成 Markdown 草稿
  -> 人审核
  -> 写入思源
```

里程碑：

- 固定一条“新知识接入提示词”；
- 能自动生成候选检索关键词；
- 能自动列出 3-10 个候选旧卡；
- 能输出插入建议。

完成标准：

- 每次接入新材料耗时小于 10 分钟；
- 大多数材料能找到旧上下文；
- 新建卡片比例下降，更新旧卡比例上升。

### 阶段 5：语义检索增强

触发条件：

- 概念卡超过 50 张；
- 来源笔记超过 100 张；
- 关键词搜索开始漏掉明显相关内容；
- 主题地图维护压力变大。

目标：

在 SQL 和反链之外增加语义召回。

可选方案：

- 使用思源技能现有的 hybrid/semantic 搜索能力；
- 接入本地 Ollama embedding；
- 使用 Qdrant 或其他向量库；
- 只索引 `10 概念卡`、`20 来源笔记`、`90 主题地图`，不全量索引原知识库。

里程碑：

- 完成卡片向量索引；
- 支持语义查询候选卡片；
- 查询结果仍然回到思源块 ID；
- 语义检索只做召回，最终上下文仍以卡片和地图为准。

完成标准：

- 同义词和近义概念能被找到；
- 语义结果没有替代结构化检索；
- 不因为向量库引入维护负担。

### 阶段 6：知识审计和自我修复

目标：

让系统能发现自身的问题。

审计项：

- 低置信度卡片；
- 长期未更新卡片；
- 没有任何反链的孤立卡片；
- 来源不足的判断；
- 互相冲突的判断；
- 主题地图没有覆盖的新概念；
- 收集箱长期未处理材料。

里程碑：

- 每周自动生成审计清单；
- 能列出待合并卡片；
- 能发现孤立卡片；
- 能提示需要补来源的判断。

完成标准：

- 知识库不会越用越乱；
- LLM 不会长期读到过时判断；
- 主题地图持续保持高价值。

## 下一步推进

当前最应该推进阶段 2：检索工作台。

优先级从高到低：

## 暂不建议做的事

- 不建议现在全量迁移旧知识库。
- 不建议现在上复杂 RAG 或向量库。
- 不建议现在用思源数据库表格视图管理所有卡片。
- 不建议为了字段完整而增加过重模板。
- 不建议把所有链接都改成块引用，只处理关键关系。

### TODO 3：增加冲突处理结构

Karpathy LLM Wiki pattern 里需要周期性发现 contradictions / stale claims。最小闭环阶段先不新增 `conflict` 类型或 relation 属性；冲突先记录在 `question` 卡或普通卡片正文的 `反证 / 风险`、`当前判断` 区域，并用思源块引用关联来源。

冲突记录建议包含：

- 冲突主题。
- 观点 A 与来源引用。
- 观点 B 与来源引用。
- 冲突类型：事实冲突 / 方案迭代 / 观点分歧 / 任务冲突。
- 当前裁决或处理建议。
- 决策来源和确认时间。

完成标准：

- 系统不会静默合并互相矛盾的判断。
- 重要冲突可以在问题卡或相关卡片正文中被展示、解决或归档。

### TODO 4：把“新知识编译流程”写入 skill 工作流

Core Workflow 已调整为持续编译视角：

```text
Detect source change
  -> Extract claims and questions
  -> Search maps and existing cards
  -> Compare existing cards
  -> Mark affected cards stale when needed
  -> Generate update proposal
  -> Human approves write
  -> Verify attributes / refs
  -> Maintain maps and audits
```

`SKILL.md` 中的执行流程：

1. 检测新增/修改材料。
2. 抽取 claims、keywords、sources、projects、open questions。
3. 先查主题地图，再查概念卡、项目卡、问题卡、来源卡。
4. 判断更新旧卡、新建卡、标记 stale、记录冲突说明。
5. 生成更新建议，列出受影响卡片、新卡、状态变化、block refs 和地图更新。
6. 用户明确确认后写入思源。
7. 设置 `custom-*` 属性。
8. 建立关键 block refs。
9. SQL 验证 attributes / refs。
10. 更新主题地图或审计清单。

完成标准：

- skill 不只是“建卡片”，而是负责维护一个会演化的 Wiki。
- 新材料能触发旧卡更新、stale 标记和冲突说明记录。

### TODO 5：升级搜索命令，增加 FTS/BM25 优先召回

当前 `siyuan-query.js search` 使用：

```sql
content LIKE '%KEYWORD%'
```

这适合 MVP 起步，但卡片变多后召回质量和性能会受限。建议增加命令：

```bash
siyuan-query.js fts-search <keyword> [--box <notebookId>] [--limit 30]
siyuan-query.js search <keyword> --mode fts|like
```

实现建议：

- 优先查询思源已有 `blocks_fts` 或 `blocks_fts_case_insensitive`。
- 如果 FTS 查询失败，再 fallback 到 `LIKE`。
- 返回字段保持和当前 search 兼容：`id`、`content`、`type`、`hpath`、`root_id`、`updated`。

完成标准：

- 新知识接入前的检索不只依赖 LIKE。
- 大量卡片下仍能快速找到候选旧卡。

## 最终判断

终局是一个基于思源块系统的个人 LLM 上下文系统。当前设计是抵达这个终局的最小可行起点，而不是临时玩具。

现在要做的不是重新设计目录，而是持续推进这条演化线：

```text
终局上下文系统
  <- 检索工作台
  <- attributes / refs / SQL
  <- 主题地图
  <- 概念卡 / 来源卡 / 项目卡
  <- 当前中文目录结构
```

所以，当前方案应该继续使用，并优先补齐检索工作台和卡片生产节奏。
